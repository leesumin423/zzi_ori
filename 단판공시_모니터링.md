# 단판공시 모니터링 (개발 진행 중 · 1단계)

동양(주)이 공시하는 **"단일판매ㆍ공급계약체결"**(현장별 공사 수주 공시)을 DART Open API로
자동 스크랩해서, **아직 회사가 관리 중인(=끝나지 않은) 현장만** 표로 정리해주는 기능입니다.
기존 [주가현황 대시보드](README.md) 안에 사이드바 "공시" 메뉴의 서브탭으로 붙어 있습니다
(공시 메뉴의 또 다른 서브탭인 지분공시는 [지분공시_모니터링.md](지분공시_모니터링.md) 참고).

> **왜 별도 문서인가**: 주가현황 대시보드는 이미 안정적으로 배포 중인 기능이라 그쪽 문서
> ([README.md](README.md), [데이터_출처.md](데이터_출처.md))는 건드리지 않았습니다. 이 기능은
> 아직 1단계(현장 스크랩)만 만들어진 개발 초기 단계라, 완성되면 자동 월간 메일링까지 붙일
> 계획입니다 — 그래서 코드/설명을 이 문서 하나로 분리해뒀습니다. **`dist/주가현황_배포.zip`
> (비개발자용 배포판)에는 아직 포함되지 않습니다.**

## 2. 코드 — 어디에 뭐가 있는가

기존 대시보드와 같은 Flask 서버(`server.py`) 안에 한 섹션으로 얹혀 있고, 별도 프로세스나
DB 없이 **요청 시점에 DART API를 스크랩 → 파싱 → 클러스터링해서 그 자리에서 JSON을
반환**하는 구조입니다.

### 파일 구성

### 처리 흐름 (`fetch_danpan_monitoring()`)

```
1. _danpan_fetch_periodic_progress()로 가장 최근 정기보고서의 "진행 현황" 표 확보
   (기준일, {신고일자(최초) 'YYYY-MM-DD', ...} 집합) — 실패 시 (None, None)
2. list.json으로 동양(주) 공시 목록 조회 (최근 10년, DANPAN_LOOKBACK_YEARS)
3. report_nm에 "단일판매" + "공급계약"이 둘 다 들어간 건만 필터링
4. 남은 건 각각 document.xml로 원문(zip 안 XML/HTML)을 받아옴
   → _danpan_parse_document()로 표 구조를 라벨 매칭해서 파싱
5. "※ 관련공시" 링크로 같은 현장의 이전/이후 공시들을 union-find로 클러스터링
6. 클러스터 안에 "해지" 공시가 있으면 그 현장 통째로 제외
7. 나머지 클러스터마다:
   - 최초 신고일자가 정기보고서 기준일 이전이면 → 그 표에 없으면 제외(관리 종료로 판단)
   - 정기보고서 기준일 이후에 새로 신고됐다면(아직 그 표에 반영될 수 없음) → 공사기간
     종료일이 지났으면 제외 (기존 방식으로 폴백)
8. 남은 클러스터마다 최초/최신 공시를 비교해 변동율 계산 후 리스트로 반환
9. 결과 + 메타(정기보고서 기준일 등)를 6시간(DANPAN_CACHE_TTL) 동안 메모리에 캐시
```

### 핵심 함수

- **`_danpan_fetch_periodic_progress(corp_code)`** — 가장 최근 사업/반기/분기보고서(정정본
  제외 — 정정본은 정정된 항목만 담고 있어 표 전체가 없을 수 있음)를 찾아 원문에서
  **"XI. 그 밖에 투자자 보호를 위하여 필요한 사항 > 1. 공시내용 진행 및 변경사항 > 나.
  단일판매ㆍ공급계약체결공시에 대한 진행 현황"** 섹션을 파싱합니다. 이 섹션은 회사가 매
  정기보고서마다 "현재 관리 중인 단판공시 현장"만 신고일자(최초) 기준으로 골라 나열한 표라,
  공사기간 종료일이 지났는지 여부보다 "아직 진행 중인가"를 훨씬 정확하게 판단할 수 있는
  근거입니다 — 공사기간이 실제로 연장돼도 정정공시를 안 내는 경우가 흔해서 종료일만으로는
  이미 끝난 현장을 걸러낼 수 없기 때문입니다. 표 제목 문단(`<p>`)을 텍스트로 찾은 뒤 그
  뒤에 나오는 두 번째 `<table>`(첫 번째는 "기준일 : YYYY년 MM월 DD일" 라벨만 있는 표)에서
  `신고일자(최초)` 열 값을 모읍니다.

- **`_danpan_parse_document(html_text)`** — 공시 원문 표에서 라벨 텍스트로 필드를 뽑습니다.
  **표의 HTML id로 찾지 않습니다** — DART 서식 프로그램 버전에 따라 같은 "단일판매ㆍ공급계약
  체결" 공시라도 표 id가 `XFormD1_Form0_Table0`(일반), `XFormD8_Form0_Table0`([기재정정]),
  `XFormD14_Form0_Table0`(2020~2022년 일부 자율공시)로 제각각이었고, 이 id를 하드코딩했다가
  `XFormD14` 서식으로 올라온 서대문 은평지사 최초 공시가 통째로 파싱 실패해 클러스터링이 깨진
  적이 있습니다(관련공시 링크로 이어져야 할 정정 건들이 고아 상태가 되어 현장 전체가 결과에서
  사라짐). 지금은 **"체결계약명"/"해지계약명"/"세부내용"(구서식) 라벨이 있는 표**를 직접
  찾아서 씁니다. [기재정정] 문서는 본문 표 앞에 "정정 전/후" 비교표가 먼저 나오는데, 계약명
  자체가 정정 대상이면 그 비교표에도 우연히 같은 라벨이 걸릴 수 있어, 여러 표가 매치되면
  **문서 뒤쪽에 나오는(=본문) 표**를 쓰도록 마지막 매치를 채택합니다. `[기재정정]` 문서에도
  본문 표는 **정정 반영 후 현재 값**으로 들어있어서 정정 전/후 비교 자체는 볼 필요가
  없습니다.

- **클러스터링 (`fetch_danpan_monitoring` 내부, union-find 부분)** — 같은 현장의 [기재정정]
  공시들을 하나로 묶는 부분입니다. 원문의 "※ 관련공시" 링크에 있는 `rcpno`는 **KRX(KIND)
  자체 접수번호**라 DART의 `rcept_no`와 자릿수 체계가 다릅니다
  (예: KRX `20220224000460` ↔ DART `20220224800460` — 날짜 8자리는 같고 뒤 6자리 일련번호의
  첫 글자만 다름). 코스피 상장사(동양)는 이 규칙("뒷자리 6자리의 첫 글자를 8로 치환")이
  일관되게 성립해서 이걸로 직접 변환해 매칭하고, 변환이 안 맞을 때만 같은 날짜에 접수된
  단판공시가 **정확히 1건**인 경우에 한해 날짜로 폴백합니다. (같은 날 서로 다른 두 현장
  공시가 겹치는 실제 사례가 있었습니다 — 2025-11-28에 광양바이오매스 정정 공시와 인천
  구월동 해지 공시가 동시에 접수됨. 후보가 여럿이면 아예 연결하지 않아 오매칭을 피합니다.)

- **필터링** — 클러스터에 "단일판매ㆍ공급계약해지" 공시가 하나라도 있으면 그 현장은 전체
  제외(더 이상 진행 중이 아니므로). 그 외에는 위 "처리 흐름" 7번의 정기보고서 기준
  판단을 씁니다. 공사기간에 종료일이 없는 경우(예: "실착공일로부터 20개월"처럼 상대 기간으로만
  적힌 계약)는 화면엔 "미정"으로 표시하되, 정기보고서 진행현황 표에 있으면 그대로 포함합니다.

- **`_fetch_annual_financials(corp_code)`** — "규정 보기" 모달에 쓸 공시의무 기준금액을
  계산합니다. `fnlttSinglAcntAll.json`(연결재무제표, `fs_div=CFS`)에서 최근 사업연도의
  `매출액`(손익계산서)과 `자산총계`(재무상태표)를 가져와, 자산총계가 2조원 이상이면
  대규모법인(2.5% 기준), 아니면 일반법인(5% 기준)으로 판단해 `매출액 × 기준비율`을 계산합니다.
  `fetch_dart_capital`과 마찬가지로 연초에 최신 사업보고서가 아직 안 나온 경우를 대비해
  최대 2개 연도 재시도하고, 결과를 24시간 캐시합니다(재무제표는 하루 안에 안 바뀌므로
  danpan 캐시보다 더 길게 잡음). 근거 조항(유가증권시장 공시규정 제7조제1항제1호다목:
  "최근 사업연도 매출액의 100분의 5(대규모법인의 경우 1,000분의 25) 이상")은 KRX
  「유가증권시장 공시·상장 업무해설서」 원문에서 직접 확인한 문구입니다.
  **연도가 바뀌면 자동으로 최신 사업연도로 갱신됩니다**: `this_year = datetime.now().year`를
  기준으로 매번 다시 계산하기 때문에, 예를 들어 2027년 3월에 2026년 사업보고서가 새로
  올라오면 그 다음 조회(늦어도 24시간 캐시 만료 후)부터 자동으로 2026년 매출액 기준으로
  바뀝니다 — 코드나 값을 손으로 갱신할 필요가 없습니다.

### 실행 / 테스트 방법

기존 대시보드와 서버가 같으므로 별도 설치는 필요 없습니다.

```bash
pip install -r requirements.txt
python server.py
```

브라우저에서 `http://localhost:5000` 접속 → 사이드바 "단판공시" 탭 클릭. 첫 조회는 정기보고서
1건 + 단판공시 원문 수십 건을 하나씩 받아오는 무거운 작업이라 **20~40초 정도 걸릴 수
있습니다** (이후 6시간은 캐시 재사용).

함수 단위로 직접 확인하고 싶으면:

```bash
python -c "import server, json; d = server.fetch_danpan_monitoring(force=True); print(json.dumps({'meta': server._danpan_cache['meta'], 'sites': d}, ensure_ascii=False, indent=2))"
```

### 사전검증 계산기 — "단판공시 대상여부 사전검증"

단판공시 표 아래에 계약(예정)일자와 계약금액을 입력하면 공시 대상 여부를 바로 알려주는
계산기가 있습니다. 사업부서가 계약 체결 전에 로그인 없이 바로 확인할 수 있게 만든 것으로,
"규정 보기"(2번 항목)가 보여주는 계산을 **원하는 날짜 기준으로** 직접 돌려볼 수 있는
버전입니다.

**계약금액은 부가세 포함 기준**입니다(사용자 확인). 입력칸은 타이핑하는 즉시 천단위
콤마가 붙습니다(0 개수를 세다 실수하는 걸 방지) — 예: `5000000000`을 치면 화면엔
`5,000,000,000`으로 보이고, 제출할 때 콤마는 자동으로 제거한 뒤 서버에 보냅니다.

**"규정 보기"와 다른 점 — 계약일자 기준 매출액**: "규정 보기"는 항상 *오늘* 기준 최근
사업연도 매출액을 씁니다. 하지만 사전검증은 과거 계약(예: 2021년부터 지금까지 이어져 온
현장들)을 그 당시 기준으로 다시 확인하는 용도로도 쓰이므로, **입력한 계약일자 시점에
실제로 참조 가능했던 매출액**을 씁니다 — 예를 들어 2023-06-02를 입력하면 "오늘(2026년)
기준 최신 매출액"이 아니라 "2023-06-02 당시 이미 제출돼 있던 가장 최근 사업보고서"(보통
전전년도, 2022년 12월 결산분)의 매출액을 찾아 씁니다.

```
GET /data?section=danpan_check&contract_date=2023-06-02&amount=49225000000
```

```json
{
  "contract_date": "2023.06.02",
  "applicable_fiscal_year": 2022,
  "applicable_report_nm": "[기재정정]사업보고서 (2022.12)",
  "applicable_report_date": "2023.04.03",
  "revenue": 767681537593,
  "assets": 983550240798,
  "is_large_corp": false,
  "threshold_pct": 0.05,
  "threshold_amount": 38384076880,
  "amount": 49225000000,
  "is_disclosure_required": true
}
```

**검증**: 광양바이오매스(2023-06-02, 492.25억), 국회대로(2021-01-14, 467.43억), 데이터센터
MEP구매(2026-06-15, 365.17억) — 실제로 전부 공시됐던 3건을 넣어봤더니 셋 다 정확히
"공시 대상"으로 판정됐습니다.

**핵심 함수**:
- **`_resolve_fiscal_year_asof(corp_code, as_of_date)`** — `list.json`으로 `pblntf_ty=A`
  (정기공시)를 조회해 `as_of_date` 이전에 접수된 것 중 report_nm이 "사업보고서"로 시작하는
  (반기ㆍ분기보고서 제외 — "최근 사업연도"는 연간 결산 기준) 가장 최근 건을 찾고, report_nm의
  `"(YYYY.MM)"` 패턴에서 사업연도를 뽑습니다.
- **`_fetch_annual_financials(corp_code, bsns_year=None)`** — 기존 "규정 보기"용 함수를
  확장했습니다. `bsns_year`를 안 주면 기존처럼 올해 기준으로 최근 사업연도를 자동 탐색하고,
  특정 연도를 주면 그 연도만 조회합니다. 캐시 키에도 연도를 포함시켜서(`(corp_code, bsns_year)`)
  서로 다른 연도 조회가 캐시를 덮어쓰지 않습니다.
- **`check_danpan_disclosure(contract_date_str, amount)`** — 위 두 함수를 엮어서 최종 판단을
  반환합니다.

### 다른 회사로 확장하려면

1. `server.py`의 `DART_CORP_CODES` 딕셔너리에 종목코드 → DART 고유번호 매핑 추가
   (DART Open API `corpCode.xml`에서 조회 가능)
2. `DANPAN_TARGET_STOCK_CODE`를 바꾸거나, 여러 회사를 동시에 보려면 `fetch_danpan_monitoring`을
   회사별로 반복 호출하도록 리팩터링 필요 (현재는 단일 회사 전용으로 짜여 있음)
3. "rcpno → rcept_no 변환 규칙"이 코스피(`corp_cls == 'Y'`) 기준으로 검증된 것이라, 코스닥
   상장사를 추가할 경우 이 변환 규칙이 그대로 맞는지 먼저 확인 필요
4. `_danpan_fetch_periodic_progress`의 표 제목 문구("단일판매"+"진행 현황")는 DART 정기보고서
   표준 서식이라 대부분 회사에 그대로 적용되지만, 만약 해당 회사가 단판공시 자체가 없는
   분기라면 정기보고서에 이 섹션이 통째로 없을 수 있음 — 이 경우 함수가 `(None, None)`을
   반환하고 종료일 기반 판단으로 자동 폴백됨

### 알려진 제한사항

- 2015년 이전 공시 일부는 `document.xml` 조회 시 `status=014`("파일이 존재하지 않습니다")로
  실패합니다 — DART 쪽 오래된 원문 아카이브 누락으로 보이며, 정기보고서 진행현황 표에도 그
  정도로 오래된 현장은 보통 이미 빠져 있어 실질 영향은 적습니다.
- 정기보고서 기준일 이후 새로 신고된 현장(위 예시의 "데이터센터 MEP구매")은 아직 정기보고서로
  검증할 수 없어 공사기간 종료일로만 판단합니다 — 종료일이 없는 "미정" 건은 다음 정기보고서가
  나올 때까지는 사람이 확인해야 합니다.
- 캐시가 서버 프로세스 메모리에만 있어서 서버를 재시작하면 초기화됩니다(다시 스크랩).
- 동양(주) 단일 회사 전용으로 하드코딩되어 있습니다 (위 "확장하려면" 참고).
