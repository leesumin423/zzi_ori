# 단판공시 모니터링 (개발 진행 중 · 1단계)

동양(주)이 공시하는 **"단일판매ㆍ공급계약체결"**(현장별 공사 수주 공시)을 DART Open API로
자동 스크랩해서, 오늘 기준 **아직 만기(공사기간)가 지나지 않은 현장만** 표로 정리해주는
기능입니다. 기존 [주가현황 대시보드](README.md) 안에 사이드바 탭 하나로 붙어 있습니다.

> **왜 별도 문서인가**: 주가현황 대시보드는 이미 안정적으로 배포 중인 기능이라 그쪽 문서
> ([README.md](README.md), [데이터_출처.md](데이터_출처.md))는 건드리지 않았습니다. 이 기능은
> 아직 1단계(현장 스크랩)만 만들어진 개발 초기 단계라, 완성되면 자동 월간 메일링까지 붙일
> 계획입니다 — 그래서 코드/설명을 이 문서 하나로 분리해뒀습니다. **`dist/주가현황_배포.zip`
> (비개발자용 배포판)에는 아직 포함되지 않습니다.**

## 2. 코드 — 어디에 뭐가 있는가

기존 대시보드와 같은 Flask 서버(`server.py`) 안에 한 섹션으로 얹혀 있고, 별도 프로세스나
DB 없이 **요청 시점에 DART API를 스크랩 → 파싱 → 클러스터링해서 그 자리에서 JSON을
반환**하는 구조입니다.

### 파일 구성

| 파일 | 역할 |
|---|---|
| [server.py](server.py) L921 이후 (`DANPAN_TARGET_STOCK_CODE`부터) | 핵심 로직 — DART 조회, 원문 파싱, 클러스터링, 필터링 |
| [server.py](server.py) `if section == 'danpan':` | `/data?section=danpan` 라우트 |
| [주가현황.html](주가현황.html) `#page-danpan` | 탭 컨테이너 + 표 마크업 |
| [dashboard.js](dashboard.js) L96\~150 | `loadDanpan()` / `renderDanpan()` — fetch 후 표 렌더링 |
| `.dart_api_key` | DART Open API 키 (레포에 이미 포함, `README.md` 참고) |

### 처리 흐름 (`fetch_danpan_monitoring()`, server.py L975)

```
1. list.json으로 동양(주) 공시 목록 조회 (최근 10년, DANPAN_LOOKBACK_YEARS)
2. report_nm에 "단일판매" + "공급계약"이 둘 다 들어간 건만 필터링
3. 남은 건 각각 document.xml로 원문(zip 안 XML/HTML)을 받아옴
   → _danpan_parse_document()로 표 구조를 라벨 매칭해서 파싱
4. "※ 관련공시" 링크로 같은 현장의 이전/이후 공시들을 union-find로 클러스터링
5. 클러스터 안에 "해지" 공시가 있으면 그 현장 통째로 제외
6. 최신 공시의 공사기간 종료일이 오늘보다 과거면 제외
7. 남은 클러스터마다 최초/최신 공시를 비교해 변동율 계산 후 리스트로 반환
8. 결과를 6시간(DANPAN_CACHE_TTL) 동안 메모리에 캐시
```

### 핵심 함수

- **`_danpan_parse_document(html_text)`** (L926) — DART 표준 공시 표(`id="XFormD1_Form0_Table0"`)를
  라벨 텍스트로 매칭해 필드를 뽑습니다. 고정된 행 번호 대신 라벨(`체결계약명`, `계약금액`,
  `시작일`/`종료일`, `계약(수주)일자`, `관련공시` 등) 기준으로 파싱하기 때문에, [기재정정]
  문서처럼 앞에 정정 요약 표가 추가로 붙어도 그대로 동작합니다. `[기재정정]` 문서에도 이
  표는 **정정 반영 후 현재 값**으로 들어있어서 정정 전/후 비교 표는 볼 필요가 없습니다.

- **클러스터링 (`fetch_danpan_monitoring` 내부, union-find 부분)** — 같은 현장의 [기재정정]
  공시들을 하나로 묶는 부분입니다. 원문의 "※ 관련공시" 링크에 있는 `rcpno`는 **KRX(KIND)
  자체 접수번호**라 DART의 `rcept_no`와 자릿수 체계가 다릅니다
  (예: KRX `20220224000460` ↔ DART `20220224800460` — 날짜 8자리는 같고 뒤 6자리 일련번호의
  첫 글자만 다름). 코스피 상장사(동양)는 이 규칙("뒷자리 6자리의 첫 글자를 8로 치환")이
  일관되게 성립해서 이걸로 직접 변환해 매칭하고, 변환이 안 맞을 때만 같은 날짜에 접수된
  단판공시가 **정확히 1건**인 경우에 한해 날짜로 폴백합니다. (같은 날 서로 다른 두 현장
  공시가 겹치는 실제 사례가 있었습니다 — 2025-11-28에 광양바이오매스 정정 공시와 인천
  구월동 해지 공시가 동시에 접수됨. 후보가 여럿이면 아예 연결하지 않아 오매칭을 피합니다.)

- **필터링** — 클러스터에 "단일판매ㆍ공급계약해지" 공시가 하나라도 있으면 그 현장은 전체
  제외(더 이상 진행 중이 아니므로). 최신 공시의 공사기간에 종료일이 없는 경우(예: "실착공일로부터
  20개월"처럼 상대 기간으로만 적힌 계약)는 만기 여부를 판단할 수 없어 **일단 진행 중으로
  간주**하고 화면엔 "미정"으로 표시합니다 — 실제로는 이미 끝났을 수 있으니 사람이 확인해야
  하는 부분입니다.

### API

```
GET /data?section=danpan          # 캐시된 결과 반환 (없으면 새로 스크랩, 6시간 캐시)
GET /data?section=danpan&refresh=1  # 캐시 무시하고 강제로 새로 스크랩
```

### 실행 / 테스트 방법

기존 대시보드와 서버가 같으므로 별도 설치는 필요 없습니다.

```bash
pip install -r requirements.txt
python server.py
```

브라우저에서 `http://localhost:5000` 접속 → 사이드바 "단판공시" 탭 클릭. 첫 조회는 원문을
수십 건씩 하나씩 받아오는 무거운 작업이라 **20~40초 정도 걸릴 수 있습니다** (이후 6시간은
캐시 재사용).

함수 단위로 직접 확인하고 싶으면:

```bash
python -c "import server, json; print(json.dumps(server.fetch_danpan_monitoring(force=True), ensure_ascii=False, indent=2))"
```

### 다른 회사로 확장하려면

1. `server.py`의 `DART_CORP_CODES` 딕셔너리에 종목코드 → DART 고유번호 매핑 추가
   (DART Open API `corpCode.xml`에서 조회 가능)
2. `DANPAN_TARGET_STOCK_CODE`를 바꾸거나, 여러 회사를 동시에 보려면 `fetch_danpan_monitoring`을
   회사별로 반복 호출하도록 리팩터링 필요 (현재는 단일 회사 전용으로 짜여 있음)
3. "rcpno → rcept_no 변환 규칙"이 코스피(`corp_cls == 'Y'`) 기준으로 검증된 것이라, 코스닥
   상장사를 추가할 경우 이 변환 규칙이 그대로 맞는지 먼저 확인 필요

### 알려진 제한사항

- 2015년 이전 공시 일부는 `document.xml` 조회 시 `status=014`("파일이 존재하지 않습니다")로
  실패합니다 — DART 쪽 오래된 원문 아카이브 누락으로 보이며, 그 정도로 오래된 공사는 이미
  만기가 지났을 가능성이 높아 실질 영향은 적습니다.
- 공사기간이 상대적 표현(예: "실착공일로부터 20개월")으로만 적힌 건은 "미정"으로 표시되고
  자동으로 만기 판단을 못합니다.
- 캐시가 서버 프로세스 메모리에만 있어서 서버를 재시작하면 초기화됩니다(다시 스크랩).
- 동양(주) 단일 회사 전용으로 하드코딩되어 있습니다 (위 "확장하려면" 참고).
