# 지분공시 모니터링 (개발 진행 중 · 1단계)

동전주 이슈로 동양(주) 임원들이 매월 자사주를 사들이고 있는데, 이걸 팀에서 DART 공시를
하나하나 찾아 엑셀로 수기 취합하고 있었습니다([협조전발송용]이 아니라 사내 참고용
`(주)동양_내부자 매매내역.xlsx`). 이 기능은 그 취합 작업을 DART Open API로 자동화해서,
**임원ㆍ주요주주별로 최근 10년간의 매수 이력을 집계**해 보여줍니다. 기존
[주가현황 대시보드](README.md) 안에 사이드바 "공시" 메뉴의 서브탭으로 붙어 있습니다
(공시 메뉴의 또 다른 서브탭인 단판공시는 [단판공시_모니터링.md](단판공시_모니터링.md) 참고).

> **왜 별도 문서인가**: [단판공시_모니터링.md](단판공시_모니터링.md)와 같은 이유입니다 —
> 아직 1단계(이력 집계)만 만들어진 개발 초기 단계라 `dist/주가현황_배포.zip`(비개발자용
> 배포판)에는 포함되지 않고, 주가현황 대시보드 자체 문서(README.md, 데이터_출처.md)와도
> 분리해뒀습니다.

(`주가현황.html` 안의 `#page-disclosures` > `#disclosure-equity`)

**"불러오는 중…" 안내 → 원문 파싱 완료 후 아래 표시:**

> 총 28명. 최근 10년간 "임원ㆍ주요주주특정증권등소유상황보고서" 중 장내매수 이력만
> 집계했습니다 (매도ㆍ증여ㆍ주식병합 등은 제외). 직위는 최신 정기보고서의 임원 현황과 공시

(2026-07-10 기준 실제 조회 결과, 총 28명 — 매일 바뀔 수 있는 값이니 스냅샷일 뿐입니다.)
각 행 "보기" 링크는 그 사람의 **최근 매수 거래가 포함된 공시 원문**으로 연결됩니다.

**검증**: 팀에서 관리하던 참고 엑셀의 임원별 "계" 행과 대조했습니다.
- 황이석: 누적 99,327주 / 평균단가 717.5원 — 참고 엑셀 99,327주 / 717.7원 (거의 일치)
- 정진학: 누적 60,014주 / 평균단가 1,733.3원 — 참고 엑셀 60,014주 / 1,733.7원 (거의 일치)

평균단가가 소수점 이하로 살짝 다른 건 DART 공시 자체가 거래별 단가를 정수(원)로 반올림해서
신고하기 때문입니다 — 참고 엑셀은 실제 매매액 합계 ÷ 수량으로 더 정밀하게 계산하는 반면,
이 기능은 DART가 신고한 단가를 그대로 가중평균하기 때문에 생기는 오차로, 실질적으로는
무시할 수준입니다.

**참고 엑셀과 다를 수 있는 점 (버그 아님)**: 참고 엑셀의 "2차(7월)" 시트에는 2026-07-06자
거래(권오정ㆍ이종면ㆍ조승범 등 다수)가 이미 기재돼 있지만, 이 기능이 만든 표에는 아직 안
잡힙니다 — 임원ㆍ주요주주 소유상황보고서는 **거래일로부터 최대 5영업일 이내**에 신고하면
되므로, 팀이 내부적으로 거래를 먼저 파악해 엑셀에 적어두더라도 DART 공식 신고는 며칠 뒤에
올라옵니다. 이 기능은 DART에 실제로 신고가 올라온 것만 집계하므로 며칠 정도 지연이 생길 수
있습니다.

**아직 없는 것 (2단계 이후 계획)**:
- 엑셀 파일 다운로드/내보내기
- 매월 자동 메일 발송
- 동양(주) 외 다른 계열사 대상 확장
- 매도ㆍ증여 등 매수 외 거래 유형 표시 (현재는 집계에서 제외만 하고 별도로 보여주진 않음)

## 2. 코드 — 어디에 뭐가 있는가

단판공시와 같은 방식(서버 프로세스 안에서 요청 시점에 DART API를 스크랩·파싱·집계해서
그 자리에서 JSON 반환, 결과는 메모리에 캐시)으로 만들어졌습니다. 정기보고서를 조회하는
로직은 단판공시와 함께 쓰기 때문에 겹치는 부분은 공용 함수로 뺐습니다.

### 파일 구성

| 파일 | 역할 |
|---|---|
| [server.py](server.py) `EQUITY_REPORT_NAME`부터 | 핵심 로직 — DART 조회, 원문 파싱, 사람별 집계 |
| [server.py](server.py) `_fetch_latest_periodic_report()` | 최근 정기보고서 원문 조회(단판공시와 공용, 6시간 캐시) |
| [server.py](server.py) `if section == 'equity':` | `/data?section=equity` 라우트 |
| [주가현황.html](주가현황.html) `#disclosure-equity` | 탭 컨테이너 + 표 마크업 |
| [dashboard.js](dashboard.js) `loadEquity()` / `renderEquity()` | fetch 후 표 렌더링 |
| [styles.css](styles.css) `#equity-table` 규칙 | 행 높이 통일(말줄임표) + 인쇄 시 페이지 분할 방지(단판공시와 공용 선택자) |

### 처리 흐름 (`fetch_equity_monitoring()`)

```
1. _equity_fetch_officer_roster()로 최근 정기보고서의 "임원 현황" 표에서
   {성명: 직위} 매핑 확보 (실패해도 진행 — 라벨링 참고용일 뿐 필수는 아님)
2. list.json으로 동양(주) 관련 지분공시(pblntf_ty=D) 목록 조회 (최근 10년)
3. report_nm이 정확히 "임원ㆍ주요주주특정증권등소유상황보고서"인 건만 필터링
   (정정본은 기존 신고서의 오기재만 바로잡는 것이라 새 거래내역이 아니므로 제외)
4. 각 건 document.xml을 받아 _equity_parse_document()로 파싱 → 성명별로 묶음
5. 사람별로 모든 공시의 "세부변동내역" 행 중 보고사유(RPT_RSN)에 "매수"가
   포함된 것만 걸러서(매도ㆍ증여ㆍ주식병합 등은 제외):
   - 최초 매수일 = 그 사람의 매수 거래일 중 최솟값
   - 최근 매수일(공시일) = 매수 거래가 포함된 공시 중 가장 최근 공시의 접수일
   - 누적 주식수 = 매수 수량 합계
   - 평균단가 = 단가가 있는 매수 거래들의 수량가중평균
   - 원문 = 그 "최근 매수" 공시의 rcept_no
6. 매수 이력이 하나도 없는 사람(매도ㆍ증여ㆍ병합만 있는 경우)은 표에서 제외
7. 최근 매수일 내림차순으로 정렬
8. 결과 + 메타를 6시간(EQUITY_CACHE_TTL) 동안 메모리에 캐시
```

### 핵심 함수

- **`_equity_parse_document(html_text)`** — "임원ㆍ주요주주 특정증권등 소유상황보고서"는
  단판공시 문서와 달리 값마다 `ACODE`(고정 항목)나 `AUNIT`+`aunitvalue`(선택 항목, 코드값)가
  직접 붙어있는 표준 XBRL형 서식이라, 라벨 텍스트 매칭 대신 이 속성으로 바로 찾습니다.
  개인 보고자든 유진기업 같은 법인 보고자든 태그 구조가 동일해서 별도 분기 없이 하나의
  파서로 처리됩니다.
  - `IFR_NM`(성명), `STF_RYN`(등기임원여부), `STF_PSM`(직위명), `MAIN_SH`(주요주주 여부)로
    보고자 정보를 읽습니다.
  - "다. 세부변동내역" 표의 각 행에서 `RPT_RSN`(보고사유, 예: "장내매수(+)")을 기준으로
    거래를 찾고, 같은 행의 `MDF_DM`(변동일), `MDF_STK_CNT`(증감 수량), `ACI_AMT2`(취득/처분
    단가)를 읽습니다. **`RPT_RSN`으로 반드시 걸러야 하는 이유**: 이 표에는 매수만 있는 게
    아니라 주식병합 같은 사유도 섞여 있고, 그 경우 `MDF_STK_CNT`가 음수로 찍혀서(예: 유진기업의
    2026-07-07 액면병합 건은 -28,784,296) 매도로 오인하거나 집계를 오염시킬 수 있습니다.
  - **`_equity_parse_num()` 숫자 파싱 관련 실제 버그**: 2018년 이전 서식은 단가 셀에
    `"1,961(원)"`처럼 단위가 붙어 나와서, 단순히 콤마만 제거하고 `int()` 변환하면 실패해
    가격이 통째로 빠지고 평균단가가 왜곡됐습니다(정진학의 2018년 5만주 매수가 평균단가
    계산에서 빠져 1,733원이 아니라 596원으로 잘못 나왔었음). 지금은 문자열 맨 앞의 숫자
    부분만 정규식(`-?[\d,]+`)으로 뽑아써서 뒤에 어떤 단위가 붙어도 정상 처리됩니다.

- **`_equity_fetch_officer_roster(corp_code)`** — 정기보고서의 "VIII. 임원 및 직원 등에 관한
  사항 > 1. 임원 및 직원 등의 현황 > 가. 임원 현황" 표에서 `SH5_NM_T`(성명)/`SH5_LEV`(직위)를
  읽어 `{성명: 직위}` 매핑을 만듭니다. 지분공시 이력에 등재된 사람이 지금도 재직 중인지,
  직위가 뭔지 참고용(주주명 셀의 title 툴팁)으로만 쓰고, 이 표에 없다고 결과에서 제외하지는
  않습니다 — 퇴임한 임원이나 주요주주 법인도 "이력"에는 남아있어야 하므로.
  [단판공시_모니터링.md](단판공시_모니터링.md)에서 "정기보고서를 1차 판단 기준으로 쓴다"고
  한 것과 달리, 여기서는 정기보고서를 **판단(제외) 기준이 아니라 라벨링 참고 자료로만**
  씁니다 — 애초에 각 소유상황보고서 문서 자체가 보고 시점의 등기여부ㆍ직위ㆍ주요주주여부를
  스스로 담고 있어서, 정기보고서 없이도 필요한 정보는 다 있기 때문입니다.

- **정기보고서 조회 공용화 (`_fetch_latest_periodic_report`)** — 단판공시와 지분공시 둘 다
  "가장 최근의 온전한(정정본 아닌) 사업/반기/분기보고서 원문 한 건"이 필요해서, 원래
  단판공시에만 있던 정기보고서 조회 로직을 공용 함수로 빼고 6시간 캐시를 붙였습니다.
  두 기능을 같은 세션에서 열어도 정기보고서를 두 번 받아오지 않습니다.

### API

```
GET /data?section=equity            # 캐시된 결과 반환 (없으면 새로 스크랩, 6시간 캐시)
GET /data?section=equity&refresh=1  # 캐시 무시하고 강제로 새로 스크랩
```

응답 형식:

```json
{
  "meta": {
    "officer_roster_available": true,
    "lookback_years": 10
  },
  "records": [
    {
      "holder_name": "황이석",
      "position": "사외이사",
      "registered_text": "등기임원",
      "is_major_shareholder": false,
      "first_buy_date": "2025-05-02",
      "latest_buy_date": "2026.06.30",
      "total_qty": 99327,
      "avg_price": 717.5,
      "disclosure_count": 15,
      "rcept_no": "20260630000445",
      "dart_url": "https://dart.fss.or.kr/dsaf001/main.do?rcpNo=20260630000445"
    }
  ]
}
```

`position`/`registered_text`/`is_major_shareholder`는 화면 표에는 별도 컬럼으로 안 보이고
주주명 셀의 title 툴팁으로만 노출됩니다(요청받은 컬럼 구성: 순번/주주명/최초 매수일/최근
매수일(공시일)/누적 주식수/평균단가/원문을 그대로 지키기 위함).

### 실행 / 테스트 방법

기존 대시보드와 서버가 같으므로 별도 설치는 필요 없습니다.

```bash
pip install -r requirements.txt
python server.py
```

브라우저에서 `http://localhost:5000` 접속 → 사이드바 "공시" → "지분공시" 서브탭 클릭. 첫
조회는 정기보고서 1건 + 소유상황보고서 원문 70여 건을 하나씩 받아오는 무거운 작업이라
**20~40초 정도 걸릴 수 있습니다** (이후 6시간은 캐시 재사용 — 단판공시와 정기보고서를
공유하므로 둘 다 열어봐도 정기보고서는 한 번만 받아옵니다).

함수 단위로 직접 확인하고 싶으면:

```bash
python -c "import server, json; d = server.fetch_equity_monitoring(force=True); print(json.dumps({'meta': server._equity_cache['meta'], 'records': d}, ensure_ascii=False, indent=2))"
```

### 다른 회사로 확장하려면

단판공시와 동일합니다 — [단판공시_모니터링.md](단판공시_모니터링.md)의 "다른 회사로
확장하려면" 참고. 이 기능은 `EQUITY_REPORT_NAME` 필터와 ACODE 기반 파싱이 DART 표준 서식에
의존하므로, 다른 코스피 상장사로 바꿔도 대체로 그대로 동작할 가능성이 높습니다(단판공시의
"rcpno→rcept_no 변환 규칙" 같은 코스피 전용 트릭에 의존하지 않기 때문).

### 알려진 제한사항

- DART 신고는 거래일로부터 최대 5영업일 걸릴 수 있어, 팀이 내부적으로 이미 파악한 아주
  최근 거래(예: 팀 엑셀엔 있지만 며칠 전 거래)가 이 표에는 아직 안 보일 수 있습니다.
  DANPAN_CACHE_TTL과 별개로 새로고침(`&refresh=1`)해도 DART에 신고 자체가 안 올라와
  있으면 나타나지 않습니다.
- 평균단가는 DART가 신고서에 반올림해 적은 거래별 단가를 그대로 가중평균한 값이라, 실제
  매매액 합계 기준 평균단가와 소수점 이하로 미세하게 다를 수 있습니다.
- 매수 이력이 전혀 없는 임원(주식을 산 적 없는 사람)은 이 표 자체에 나타나지 않습니다 —
  "지분공시 이력"이 목적이라 매수 기록이 없으면 보여줄 이력이 없기 때문입니다.
- 캐시가 서버 프로세스 메모리에만 있어서 서버를 재시작하면 초기화됩니다(다시 스크랩).
- 동양(주) 단일 회사 전용으로 하드코딩되어 있습니다.
