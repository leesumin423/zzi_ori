# 지분공시 모니터링 (개발 진행 중 · 1단계)

동전주 이슈로 동양(주) 임원들이 매월 자사주를 사들이고 있는데, 이걸 팀에서 DART 공시를
하나하나 찾아 엑셀로 수기 취합하고 있었습니다([협조전발송용]이 아니라 사내 참고용
`(주)동양_내부자 매매내역.xlsx`). 이 기능은 그 취합 작업을 DART Open API로 자동화해서,
**임원ㆍ주요주주별로 최근 10년간의 매수 이력을 집계**해 보여줍니다. 기존
[주가현황 대시보드](README.md) 안에 사이드바 "공시" 메뉴의 서브탭으로 붙어 있습니다
(공시 메뉴의 또 다른 서브탭인 단판공시는 [단판공시_모니터링.md](단판공시_모니터링.md) 참고).

> **왜 별도 문서인가**: [단판공시_모니터링.md](단판공시_모니터링.md)와 같은 이유입니다 —
> 아직 1단계(이력 집계)만 만들어진 개발 초기 단계라 `dist/주가현황_배포.zip`(비개발자용
> 배포판)에는 포함되지 않고, 주가현황 대시보드 자체 문서(README.md, 데이터_출처.md)와도
> 분리해뒀습니다.

**"불러오는 중…" 안내 → 원문 파싱 완료 후 아래 표시:**

> 총 15명. 최근 10년간 "임원ㆍ주요주주특정증권등소유상황보고서" 중 장내매수 이력만
> 집계했습니다 (매도ㆍ증여ㆍ주식병합 등은 제외). 현재 정기보고서의 임원 현황에 없는(=퇴임한)

각 행 "보기" 링크는 그 사람의 **최근 매수 거래가 포함된 공시 원문**으로 연결됩니다.

표 제목 옆 **"규정 보기"** 버튼을 누르면 임원ㆍ주요주주 소유상황 보고의무를 도식으로 보여주는

5일 이내 무조건 보고해야 하는 자본시장법 제173조 의무라는 점을 함께 설명합니다.

**검증**: 팀에서 관리하던 참고 엑셀의 임원별 "계" 행과 대조했습니다.
- 황이석: 누적 99,327주 / 평균단가 717.5원 — 참고 엑셀 99,327주 / 717.7원 (거의 일치)
- 정진학: 누적 60,014주 / 평균단가 1,733.3원 — 참고 엑셀 60,014주 / 1,733.7원 (거의 일치)
- 안준수: 평균단가 627.2원, 황부익: 평균단가 1,094.0원 — 둘 다 참고 엑셀과 정확히 일치
  (2023년 "신규임원시" 최초 보고분까지 정확히 포함된 이후)
- 유진기업(주): 사용자 확인으로 실제 버그 2건을 찾아 고친 뒤 누적 주식수가 85,000주 →
  28,784,296주로 정정되고, 2026-06-24 800,000주 취득 건도 최근 매수일에 정확히 반영됨
  (자세한 원인은 "코드" 섹션 참고)

평균단가가 소수점 이하로 살짝 다른 건 DART 공시 자체가 거래별 단가를 정수(원)로 반올림해서
신고하기 때문입니다 — 참고 엑셀은 실제 매매액 합계 ÷ 수량으로 더 정밀하게 계산하는 반면,
이 기능은 DART가 신고한 단가를 그대로 가중평균하기 때문에 생기는 오차로, 실질적으로는
무시할 수준입니다.

**"누적 주식수" 기준 (중요, 사용자 피드백으로 수정)**: 처음엔 매수 거래 수량을 다 더해서
계산했는데, 사용자 확인 결과 이러면 값이 들쭉날쭉했습니다 — 어떤 사람은 보고서에 매수
거래가 여러 건 몰려 있고, 어떤 사람은 증여ㆍ주식병합 등 매수가 아닌 변동도 섞여 있어서
"매수 합계"가 실제 보유수량과 다를 수 있었기 때문입니다. 그래서 지금은 **가장 최근 공시에
찍힌 "변동후" 최종 보유수량**을 그대로 씁니다(아래 "코드" 섹션의 `final_holding` 참고) —
결국 딱 하나의 확정값만 신뢰하는 방식입니다.

**참고 엑셀과 다를 수 있는 점 (버그 아님)**: 참고 엑셀의 "2차(7월)" 시트에는 2026-07-06자
거래(권오정ㆍ이종면ㆍ조승범 등 다수)가 이미 기재돼 있지만, 이 기능이 만든 표에는 아직 안
잡힙니다 — 임원ㆍ주요주주 소유상황보고서는 **거래일로부터 5일 이내**에 신고하면 되므로,
팀이 내부적으로 거래를 먼저 파악해 엑셀에 적어두더라도 DART 공식 신고는 며칠 뒤에 올라옵니다.
이 기능은 DART에 실제로 신고가 올라온 것만 집계하므로 며칠 정도 지연이 생길 수 있습니다.

**아직 없는 것 (2단계 이후 계획)**:
- 엑셀 파일 다운로드/내보내기
- 매월 자동 메일 발송
- 동양(주) 외 다른 계열사 대상 확장
- 매도ㆍ증여 등 매수 외 거래 유형 표시 (현재는 집계에서 제외만 하고 별도로 보여주진 않음)

## 2. 코드 — 어디에 뭐가 있는가

단판공시와 같은 방식(서버 프로세스 안에서 요청 시점에 DART API를 스크랩·파싱·집계해서
그 자리에서 JSON 반환, 결과는 메모리에 캐시)으로 만들어졌습니다. 정기보고서를 조회하는
로직은 단판공시와 함께 쓰기 때문에 겹치는 부분은 공용 함수로 뺐습니다.

### 파일 구성

| 파일 | 역할 |
|---|---|
| [server.py](server.py) `EQUITY_REPORT_NAME`부터 | 핵심 로직 — DART 조회, 원문 파싱, 사람별 집계 |
| [server.py](server.py) `_fetch_latest_periodic_report()` | 최근 정기보고서 원문 조회(단판공시와 공용, 6시간 캐시) |
| [server.py](server.py) `if section == 'equity':` | `/data?section=equity` 라우트 |
| [주가현황.html](주가현황.html) `#disclosure-equity` | 탭 컨테이너 + 표 마크업 |
| [주가현황.html](주가현황.html) `#ruleModalOverlay` | "규정 보기" 모달 (단판공시ㆍ지분공시 공용) |
| [dashboard.js](dashboard.js) `loadEquity()` / `renderEquity()` | fetch 후 표 렌더링 |
| [dashboard.js](dashboard.js) `showRuleModal()` / `ruleEquityHtml()` | 규정 모달 내용 렌더링 (정적 텍스트, 재무 데이터 계산 없음) |
| [styles.css](styles.css) `#equity-table` 규칙 | 행 높이 통일(말줄임표) + 인쇄 시 페이지 분할 방지(단판공시와 공용 선택자) |

### 처리 흐름 (`fetch_equity_monitoring()`)

```
1. _equity_fetch_officer_roster()로 최근 정기보고서의 "임원 현황" 표에서
   {성명: {직위, 등기구분(사내이사/사외이사/미등기)}} 매핑 확보 — 조회 자체에
   실패하면 None(이 경우 7번의 퇴임자 필터링은 건너뛴다. 실패했는데 필터링을
   강행하면 전원이 "퇴임자"로 잘못 걸러지기 때문)
2. list.json으로 동양(주) 관련 지분공시(pblntf_ty=D) 목록 조회 (최근 10년)
3. report_nm이 정확히 "임원ㆍ주요주주특정증권등소유상황보고서"인 건만 필터링
   (정정본은 기존 신고서의 오기재만 바로잡는 것이라 새 거래내역이 아니므로 제외)
4. 각 건 document.xml을 받아 _equity_parse_document()로 파싱 → 성명별로 묶음
5. 사람별로 모든 공시의 "세부변동내역" 행 중 보고사유(RPT_RSN)에 "매수"가
   포함된 것만 걸러서(매도ㆍ증여ㆍ주식병합 등은 제외):
   - 최초 매수일 = 그 사람의 매수 거래일 중 최솟값
   - 최근 매수일(공시일) = 매수 거래가 포함된 공시 중 가장 최근 공시의 접수일
   - 누적 주식수 = 전체 공시 중 가장 최근 것의 최종 보유수량(final_holding) —
     매수 거래만 다시 더하지 않고, 그 시점에 확정된 값 하나만 신뢰
   - 평균단가 = 단가가 있는 매수 거래들의 수량가중평균
   - 원문 = 그 "최근 매수" 공시의 rcept_no
6. 매수 이력이 하나도 없는 사람(매도ㆍ증여ㆍ병합만 있는 경우)은 표에서 제외
7. 주요주주(법인 등)가 아니면서 현재 임원 현황(1번)에 없는 사람은 제외(퇴임자)
8. role_label 결정: 주요주주면 "최대주주", 아니면 1번 roster의 등기구분
   (사내이사/사외이사/미등기) — roster에 없는 예외적인 경우만 공시 원문 자체의
   등기임원여부로 대체. "OO 주식회사" 이름은 "OO(주)"로 정리
9. 주주명(가나다순)으로 정렬
10. 결과 + 메타를 6시간(EQUITY_CACHE_TTL) 동안 메모리에 캐시
```

### 핵심 함수

- **`_equity_parse_document(html_text)`** — "임원ㆍ주요주주 특정증권등 소유상황보고서"는
  단판공시 문서와 달리 값마다 `ACODE`(고정 항목)나 `AUNIT`+`aunitvalue`(선택 항목, 코드값)가
  직접 붙어있는 표준 XBRL형 서식이라, 라벨 텍스트 매칭 대신 이 속성으로 바로 찾습니다.
  개인 보고자든 유진기업 같은 법인 보고자든 태그 구조가 동일해서 별도 분기 없이 하나의
  파서로 처리됩니다.
  - `IFR_NM`(성명), `STF_RYN`(등기임원여부), `STF_PSM`(직위명), `MAIN_SH`(주요주주 여부)로
    보고자 정보를 읽습니다.
  - "다. 세부변동내역" 표의 각 행에서 `STR_KND`(증권종류)가 **보통주(코드 "1")가 아니면
    그 행은 통째로 건너뜁니다** — 유진기업처럼 보통주와 우선주를 모두 보유한 보고자는 한
    공시 안에 증권종류별로 행이 나뉘어 나오는데(우선주는 "2"), 이 필드를 안 보고 처리하면
    우선주 수량이 보통주 수량과 섞이거나 덮어씁니다(아래 `final_holding` 버그 사례 참고).
    이 기능은 동전주(보통주) 매수 이력만 보는 게 목적이라 우선주는 애초에 대상이 아닙니다.
    남은 각 행에서 `RPT_RSN`(보고사유), `MDF_DM`(변동일), `MDF_STK_CNT`(증감 수량),
    `AFR_STK_CNT`(변동후 수량), `ACI_AMT2`(취득/처분단가)를 읽습니다.
  - **`final_holding` (누적 주식수의 실제 근거) — 실제 버그를 고치며 정착한 로직**: 처음엔
    "합계" 행의 변동후 합계(`AFR_STK_SUM`)를 썼는데, 받아보니 이 값은 **거래가 1건뿐인
    보고서에서만** 채워지고 여러 건이면 "-"로 비어있는 서식이었습니다(정진학의 2026-06-10
    보고서는 매수가 2건이라 합계 행 변동후가 "-", 파싱하면 None). 지금은 "합계" 행 대신
    **표 맨 마지막 개별 거래 행의 변동후(`AFR_STK_CNT`)**를 씁니다(표는 항상 날짜
    오름차순이라 마지막 행이 곧 최신). 여기서 두 번째 실제 버그가 발견됐다 — 유진기업의
    2026-07-07 주식병합 공시가 **보통주 행**(변동후 28,784,296주)과 **우선주 행**(변동후
    85,000주) 두 개로 나뉘어 있었는데, `STR_KND` 필터링 없이 "마지막 행"만 가져오다 보니
    우선주 행(85,000)을 누적 주식수로 잘못 채택했었다. 위 `STR_KND` 필터로 우선주 행 자체를
    걸러내면서 함께 해결됨(유진기업 누적 주식수: 85,000 → 28,784,296으로 정정).
  - **`_equity_parse_num()` 숫자 파싱 관련 버그**: 2018년 이전 서식은 단가 셀에
    `"1,961(원)"`처럼 단위가 붙어 나와서, 단순히 콤마만 제거하고 `int()` 변환하면 실패해
    가격이 통째로 빠지고 평균단가가 왜곡됐습니다. 지금은 문자열 맨 앞의 숫자 부분만
    정규식(`-?[\d,]+`)으로 뽑아써서 뒤에 어떤 단위가 붙어도 정상 처리됩니다.

- **`_equity_is_buy(t)` — "매수 이력" 판단 기준, 세 번째 실제 버그**: 처음엔 보고사유
  텍스트에 "매수"가 들어간 것만 걸렀는데("장내매수(+)" 등), 유진기업의 2026-06-24
  800,000주 취득은 사유가 **"시간외매매(+)"**였고 안준수ㆍ황부익이 신규 임원이 되면서
  최초로 보고한 지분은 사유가 **"신규보고(+)"**였습니다 — 둘 다 "매수"라는 단어가 없어서
  누락되고 있었습니다(유진기업의 800,000주 매수 내역이 통째로 빠진 것도 이 때문). 지금은
  사유 텍스트를 화이트리스트로 매칭하는 대신 **"수량이 늘었고(`qty>0`) 단가가 있다
  (`price is not None`)"**로 판단합니다 — 무상증자ㆍ주식병합ㆍ상속ㆍ증여처럼 대가 없이
  수량만 바뀌는 사유는 취득/처분단가란이 항상 "-"(파싱하면 None)라 이 조합으로 자연스럽게
  제외됩니다. 이 수정으로 안준수ㆍ황부익의 평균단가도 참고 엑셀과 정확히 일치하게 됐습니다
  (안준수 627.2원, 황부익 1,094.0원).

- **`_equity_fetch_officer_roster(corp_code)`** — 정기보고서의 "VIII. 임원 및 직원 등에 관한
  사항 > 1. 임원 및 직원 등의 현황 > 가. 임원 현황" 표에서 `SH5_NM_T`(성명)/`SH5_LEV`(직위)
  뿐 아니라 `SH5_REG_DRCT`(등기구분: "사내이사"/"사외이사"/"미등기")도 읽어
  `{성명: {position, reg_type}}` 매핑을 만듭니다. 표 앞쪽에 "최대주주 및 특수관계인" 같은
  다른 표에서 우연히 같은 `SH5_NM_T` 태그가 걸리는 행들이 있는데, 그 행들은
  `SH5_REG_DRCT`가 없어서(직위 정보 자체가 없는 행) 자동으로 걸러집니다. **이 표에 없는
  사람(=현재 임원이 아닌 사람)은 결과에서 제외됩니다** — 최근 10년 이력이라 2016~2020년
  사이 퇴임한 임원들까지 계속 나오는 문제가 있었는데(사용자 지적으로 확인), 이 표로
  "현재 재직 중"인지 검증해 걸러냅니다. 단, **주요주주(법인 등, `is_major_shareholder=True`)는
  애초에 이 표 대상이 아니므로 걸러내지 않습니다**(유진기업 같은 경우). 정기보고서 조회
  자체가 실패하면 `None`을 반환하고, 이 경우 호출부는 필터링을 아예 건너뜁니다(실패를
  "전원 퇴임"으로 오판하지 않도록).

- **`role_label` / 표시용 이름 정리 (`fetch_equity_monitoring` 내부)** — 화면 주주명 셀에
  "이름(등기구분)" 형태로 보여주기 위한 태그를 만듭니다: 주요주주면 `"최대주주"`, 아니면
  roster의 `reg_type`(사내이사/사외이사/미등기)을 그대로 씁니다. "OO 주식회사"로 끝나는
  이름은 "OO(주)"로 바꿔서(예: "유진기업 주식회사" → "유진기업(주)") 팀에서 익숙한 표기로
  맞췄습니다. 최종 정렬은 이 표시용 이름 기준 가나다순입니다.

- **정기보고서 조회 공용화 (`_fetch_latest_periodic_report`)** — 단판공시와 지분공시 둘 다
  "가장 최근의 온전한(정정본 아닌) 사업/반기/분기보고서 원문 한 건"이 필요해서, 원래
  단판공시에만 있던 정기보고서 조회 로직을 공용 함수로 빼고 6시간 캐시를 붙였습니다.
  두 기능을 같은 세션에서 열어도 정기보고서를 두 번 받아오지 않습니다.

### API

```
GET /data?section=equity            # 캐시된 결과 반환 (없으면 새로 스크랩, 6시간 캐시)
GET /data?section=equity&refresh=1  # 캐시 무시하고 강제로 새로 스크랩
```

응답 형식:

```json
{
  "meta": {
    "officer_roster_available": true,
    "lookback_years": 10
  },
  "records": [
    {
      "holder_name": "황이석",
      "role_label": "사외이사",
      "position": "사외이사",
      "registered_text": "등기임원",
      "is_major_shareholder": false,
      "first_buy_date": "2025.05.02",
      "latest_buy_date": "2026.06.30",
      "total_qty": 99327,
      "avg_price": 717.5,
      "disclosure_count": 15,
      "rcept_no": "20260630000445",
      "dart_url": "https://dart.fss.or.kr/dsaf001/main.do?rcpNo=20260630000445"
    }
  ]
}
```

`holder_name`은 "이름" 컬럼, `role_label`은 "구분" 컬럼에 각각 그대로 들어갑니다(처음엔
"이름(구분)"으로 한 셀에 합쳐서 보여줬는데, 사용자 요청으로 별도 컬럼 두 개로 나눴습니다).
`position`/`registered_text`/`is_major_shareholder`는 별도 컬럼 없이 API 응답에만 남아있는
원본 참고용 필드입니다.

### 실행 / 테스트 방법

기존 대시보드와 서버가 같으므로 별도 설치는 필요 없습니다.

```bash
pip install -r requirements.txt
python server.py
```

브라우저에서 `http://localhost:5000` 접속 → 사이드바 "공시" → "지분공시" 서브탭 클릭. 첫
조회는 정기보고서 1건 + 소유상황보고서 원문 70여 건을 하나씩 받아오는 무거운 작업이라
**20~40초 정도 걸릴 수 있습니다** (이후 6시간은 캐시 재사용 — 단판공시와 정기보고서를
공유하므로 둘 다 열어봐도 정기보고서는 한 번만 받아옵니다).

함수 단위로 직접 확인하고 싶으면:

```bash
python -c "import server, json; d = server.fetch_equity_monitoring(force=True); print(json.dumps({'meta': server._equity_cache['meta'], 'records': d}, ensure_ascii=False, indent=2))"
```

### 다른 회사로 확장하려면

단판공시와 동일합니다 — [단판공시_모니터링.md](단판공시_모니터링.md)의 "다른 회사로
확장하려면" 참고. 이 기능은 `EQUITY_REPORT_NAME` 필터와 ACODE 기반 파싱이 DART 표준 서식에
의존하므로, 다른 코스피 상장사로 바꿔도 대체로 그대로 동작할 가능성이 높습니다(단판공시의
"rcpno→rcept_no 변환 규칙" 같은 코스피 전용 트릭에 의존하지 않기 때문).

### 알려진 제한사항

- DART 신고는 거래일로부터 최대 5영업일 걸릴 수 있어, 팀이 내부적으로 이미 파악한 아주
  최근 거래(예: 팀 엑셀엔 있지만 며칠 전 거래)가 이 표에는 아직 안 보일 수 있습니다.
  새로고침(`&refresh=1`)해도 DART에 신고 자체가 안 올라와 있으면 나타나지 않습니다.
- **누적 주식수는 "임원ㆍ주요주주 소유상황보고서" 기준 최종값이라, 이 보고서 유형으로
  잡히지 않는 지분 변동(예: 대량보유상황보고서로만 신고되는 경우)은 반영되지 않습니다.**
  유진기업의 경우 이 보고서 유형으로 최근까지(2026년) 신고가 계속 이어지고 있어서 실제
  값과 잘 맞지만, 만약 어떤 주요주주가 이 보고서 유형 자체를 오래 안 냈다면(예: 대량보유
  상황보고서로만 신고를 이어간 경우) 표시된 누적 주식수가 실제 현재 보유량과 다를 수
  있습니다 — 대주주의 정확한 보유 현황은 이 표가 아니라 대량보유상황보고서나 정기보고서의
  "최대주주 현황"을 봐야 합니다.
- **보통주만 집계합니다.** 우선주 등 다른 증권종류로 보고된 수량은 제외됩니다(이 기능의
  목적이 동전주=보통주 매수 모니터링이기 때문).
- 평균단가는 DART가 신고서에 반올림해 적은 거래별 단가를 그대로 가중평균한 값이라, 실제
  매매액 합계 기준 평균단가와 소수점 이하로 미세하게 다를 수 있습니다.
- 매수 이력이 전혀 없는 임원(주식을 산 적 없는 사람)은 이 표 자체에 나타나지 않습니다 —
  "지분공시 이력"이 목적이라 매수 기록이 없으면 보여줄 이력이 없기 때문입니다.
- 캐시가 서버 프로세스 메모리에만 있어서 서버를 재시작하면 초기화됩니다(다시 스크랩).
- 동양(주) 단일 회사 전용으로 하드코딩되어 있습니다.
