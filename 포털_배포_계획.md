## 전제

- **대상 인원**: 자금팀 내부 최소 3명(상시), 임원 최대 4명(가끔 열람) — 단판공시는 **사업부서**도
  봐야 함(계약 체결 전 공시대상 여부를 사전 검증하는 용도로).
- **연동 방식**: 외부 포털이 아니라 **사내 그룹웨어 포털**에 연결. 외부 인터넷 노출 없음.
- 지분공시(임원 개인 매매내역)는 민감정보라 사업부서까지 열리면 안 됨 — **탭 단위 접근 제한**이
  실질적으로 필요.

## 필요한 것 (카테고리별)

### 1. 상시 구동 서버
- 지금은 개발자 PC에서 `python server.py`를 켜둬야만 동작 — 사내망에 **상시 켜져있는 서버**(유휴
  PC, 사내 가상서버 등) 필요.
- `app.run(debug=True)`(Flask 개발 서버)는 동시접속·안정성이 약함 → **waitress**(Windows용
  프로덕션 WSGI 서버)로 교체. 코드 몇 줄 수준.
- 서버 PC 재부팅 시 자동 시작되도록 Windows 서비스 또는 작업 스케줄러 등록.

### 2. 그룹웨어 연동 방식 (그룹웨어 제품 확인 필요 — 착수 전 담당자 문의)
난이도 순:
1. **링크만 추가**: 그룹웨어 메뉴에 사내망 URL(`http://내부서버IP:포트`) 링크 추가 — 그룹웨어
   로그인과는 별개로 열림. 제일 간단.
2. **iframe 삽입**: 그룹웨어 화면 안에 埋め込み — 그룹웨어가 iframe을 허용하는지 확인 필요(보안
   정책상 막아둔 곳 많음). 그룹웨어가 HTTPS라면 http 콘텐츠가 차단될 수 있어 확인 필요.
3. **SSO 연동**: 그룹웨어 로그인 세션을 그대로 인정 — 매끄럽지만 구현 난이도 있음. 나중 단계로
   미뤄도 됨.

### 3. 접근 제어
- 인원이 매우 적으므로(최대 7명) **사내망 IP 대역 제한**(방화벽에서 외부·타 부서 IP 차단)만으로도
  1차로 충분할 수 있음.
- **탭 단위 권한 분리 필요**: 단판공시(사업부서 공유)와 지분공시(자금팀 전용, 임원 개인정보 포함)를
  분리해야 함 — 최소한 지분공시 탭에 별도 비밀번호나 로그인을 붙이는 정도의 작업이 필요(현재는
  권한 분리 기능 없음, 신규 구현 필요).

### 4. 사전검증(공시대상 판단) 기능
- **완료**: 단판공시 탭에 계약일자·계약금액을 입력하면 그 계약일자 시점 기준 매출액으로
  공시대상 여부를 판단하는 계산기를 만들었습니다 (`/data?section=danpan_check`). 사업부서든
  누구든 로그인 없이 바로 씁니다. 자세한 내용은 [단판공시_모니터링.md](단판공시_모니터링.md)
  "사전검증 계산기" 항목 참고.
- 그룹웨어 이관 시에도 이 기능은 그대로 옮기면 됨 — 별도 추가 작업 불필요.

### 5. 기타
- DART API 키 사용량: 인원이 적어(최대 7명) 지금의 캐시(단판공시·지분공시 6시간, 재무제표
  24시간)로 충분히 감당됨 — 별도 조치 불필요.
- HTTPS: 사내망 전용이면 필수는 아니지만, 그룹웨어가 HTTPS를 쓰면 iframe 삽입 시 http 콘텐츠
  차단 이슈 있을 수 있음(위 2번 참고).

## 6. 실제 포털(tyia.vercel.app) 연동 — 추가 검토사항 (2026-07-13)

기존 위 전제(1~5번)는 "사내 그룹웨어에 연결"을 가정하고 논의한 것인데, 실제로는 회원님이
이미 자체 포털을 만들어 운영 중이었습니다: **`https://tyia.vercel.app`, "(주)동양 내부회계
PORTAL"**. 여기에 주가현황ㆍ공시를 위쪽 카테고리(컬럼) 중 하나로 얹어서 팀에 공유하고
싶다는 것이 이번 요청입니다. 로그인 화면까지만 확인했고(사번 계정이 없어 대시보드 내부는
직접 못 들어가봄), 확인된 사실과 그로 인해 **바뀌는 전제**, 그리고 **통합에 필요한 작업**을
정리합니다.

### 확인된 사실 (로그인 화면 + 네트워크 요청 기준)

- **호스팅**: Vercel에 배포된 **Vite + React SPA**(정적 빌드 산출물 배포, `rolldown-runtime`ㆍ
  `vendor-react`ㆍ`vendor-charts` 등 청크 확인됨 — 이미 자체 차트 라이브러리도 씀).
- **인증/데이터베이스**: **Supabase**(`vendor-supabase`, `useAuth` 청크 확인) — 자체 회원
  테이블에 사번을 아이디로 매핑해 로그인하는 커스텀 인증으로 추정(Supabase 기본 이메일
  로그인이 아니라 "사번 + 초기비번 ty+사번" 방식).
- **로그인 게이트**: 사번(직원번호) + 비밀번호. `/dashboard` 등 내부 경로에 로그인 없이
  접근하면 로그인 화면으로 리다이렉트됨.
- **상단 카테고리(컬럼) 구성**: 로그인 화면에 이미 4개 사업영역이 보임 — 레미콘 / 건설 /
  인프라 / 금융("Construction Materials Network"/"Construction Services"/
  "Infrastructure Engineering"/"Finance & ICFR Tech"). 회원님이 말한 "위 컬럼"이 이 구조로
  보이고, 주가현황ㆍ공시는 이 중 하나(예: 금융)에 속하거나 별도 카테고리로 추가하는 방식이
  될 것 같습니다.
- **인터넷 노출 여부**: 그룹웨어처럼 사내망 전용이 아니라 **퍼블릭 인터넷에 떠 있는 서비스**
  (Vercel URL, 사번 로그인만으로 접근 통제). → 기존 계획의 "3. 접근 제어"에 있던 "사내망 IP
  대역 제한으로 1차 방어"라는 전제가 **여기엔 적용되지 않습니다.** 보안은 전적으로 이 포털의
  로그인/권한 시스템에 의존하게 됩니다.

### 통합 방식 두 가지 옵션

**옵션 A. 포털의 React 라우트로 직접 편입 (권장 — 단, 작업량 큼)**
- 지금의 서버(Flask/`server.py`)가 하는 "DART 스크래핑 → 파싱 → 집계" 로직을, 브라우저가
  직접 조회하는 대신 **주기적으로 실행해 Supabase 테이블에 결과를 적재하는 배치 작업**으로
  바꾸고(예: GitHub Actions cron, 또는 별도 스케줄러가 몇 시간마다 Python 스크립트 실행 →
  Supabase에 upsert), 포털의 React 프런트엔드는 그 테이블을 Supabase 클라이언트로 그대로
  조회해서 렌더링.
- 장점: 포털의 기존 로그인/권한 체계를 자연스럽게 그대로 씀(탭별 권한 분리도 Supabase RLS
  정책으로 해결 가능 — 지분공시 테이블은 자금팀 role만 select 가능하도록). 서버를 상시
  띄워둘 필요도 없어짐(배치 스크립트만 정기 실행하면 됨). UI 디자인도 포털과 통일됨(지금의
  바닐라 HTML/CSS를 포털의 React 컴포넌트 스타일로 새로 짜야 함).
- 단점: 작업량이 가장 큼 — (1) 현재 파싱ㆍ집계 로직(`server.py`)은 그대로 재사용 가능하지만
  Flask 라우트 대신 배치 스크립트 형태로 감싸야 함, (2) 화면(HTML/CSS/JS)은 포털의 React
  컴포넌트로 다시 만들어야 함(1:1 이식이 아니라 재작성에 가까움), (3) Supabase 테이블
  스키마 설계, RLS 정책 작성이 새로 필요함, (4) 단판공시 사전검증 계산기처럼 "그 자리에서
  즉시 계산"하는 기능은 별도의 API 엔드포인트(Supabase Edge Function 등)로 옮겨야 함.

**옵션 B. 지금 앱을 그대로 두고 iframe으로 삽입 (빠름 — 임시 방편에 가까움)**
- `server.py`를 상시 구동 가능한 곳(사내 서버든 소규모 클라우드 VM이든)에 HTTPS로 올려두고,
  포털의 새 라우트/탭 안에 `<iframe src="https://내부서버주소">`로 삽입.
- 장점: 지금 만든 화면ㆍ로직을 거의 그대로 재사용 — 추가 개발이 가장 적음.
- 단점:
  - **HTTPS 필수**: 포털이 HTTPS(Vercel)이므로 iframe 안에 http 콘텐츠를 넣으면 브라우저가
    차단함(mixed content) — Flask 앞에 리버스 프록시(예: Caddy)로 HTTPS 인증서를 씌우거나,
    Cloudflare Tunnel/ngrok 같은 방식으로 HTTPS 엔드포인트를 만들어야 함.
  - **디자인 이질감**: 지금 화면은 포털과 다른 CSS/톤으로 보여서 "얹어놓은 느낌"이 남음.
  - **이중 인증 체계**: 포털 로그인과 이 앱은 서로 다른 인증이라, 포털에 로그인했다고 이 앱도
    자동 인가되는 게 아님 — 최소한 지분공시 탭에는 별도 보호(간단한 공유 비밀번호나, 포털이
    발급하는 토큰을 iframe URL에 실어 보내고 서버가 검증하는 정도)가 필요.
  - **CORS/보안 헤더**: iframe으로 삽입 시 `X-Frame-Options`/`Content-Security-Policy:
    frame-ancestors`를 포털 도메인만 허용하도록 설정해야 함(현재 Flask 앱엔 이런 헤더 설정이
    없음 — 없으면 다른 사이트도 iframe으로 끌어다 쓸 수 있어 위험).

### 결정 필요 사항 (착수 전 확인)

1. 옵션 A(포털 네이티브 편입) vs 옵션 B(iframe 임베드) — 작업량과 유지보수 방식 차이를
   보고 방향 결정.
2. 옵션 A라면: 포털 Supabase 프로젝트에 새 테이블(단판공시/지분공시/대량보유상황보고서 각각)을
   추가할 권한과, 배치 스크립트를 정기 실행할 위치(회원님 PC 작업 스케줄러 vs GitHub Actions
   vs 별도 서버)를 정해야 함.
3. 옵션 B라면: Flask를 상시 구동할 위치(기존 4번 항목과 동일) + HTTPS 확보 방법을 먼저
   정해야 함.
4. 어느 옵션이든: 포털의 4개 카테고리(레미콘/건설/인프라/금융) 중 어디에 넣을지, 아니면 새
   카테고리를 만들지 — 회원님 확인 필요.
5. 지분공시(임원 개인 매매내역)의 자금팀 전용 접근 제한을, 포털이 이미 갖고 있을 역할(role)
   체계로 처리할 수 있는지 확인 필요(옵션 A라면 Supabase RLS로 자연스럽게 해결될 가능성
   높음).

### 옵션 A 확정 (2026-07-13) — Flask 서버가 불안정(뻗음)해서 상시 구동 자체가 부적합하다고
판단, **옵션 A(Supabase 배치 적재 + 포털 네이티브 편입)로 방향 확정**. UI도 포털 디자인에
맞춰 새로 만들기로 함. 포털 소스코드 접근은 없이 **화면만 보고 색상ㆍ폰트ㆍ레이아웃을
최대한 비슷하게 재현**하는 방식으로 진행(완전히 동일한 컴포넌트 재사용은 아님 — 코드 없이
따라 그리는 한계가 있다는 점 감안).

**디자인 목업 완료 (2026-07-13)**: 회원님이 공유해주신 포털 내부 화면 스크린샷(증빙관리ㆍ
KPI(가상)ㆍAI챗봇 3개 페이지) 기준으로 색상ㆍ타이포ㆍ카드/배지/표 컴포넌트를 그대로 재현한
"공시관리" 탭 목업을 만들어 확인받았습니다 — 단판공시(7건)ㆍ지분공시(16명)ㆍ대량보유상황
보고서(19명·법인, 보고자명 rowspan 병합 포함) 실제 데이터로 채운 정적 HTML입니다. 실제
React 컴포넌트로 옮길 때 이 목업의 색상 토큰ㆍ레이아웃 구조를 그대로 기준으로 삼으면 됩니다.

**착수를 위해 회원님께 받아야 하는 것 (2가지)**:
1. **Supabase 프로젝트 URL + service role key** — 새 테이블(단판공시/지분공시/대량보유상황
   보고서) 설계ㆍ생성과, 배치 스크립트가 그 테이블에 쓰기 위해 필요. **service role key는

### 계열사+공시현황 축소판 포털 레이아웃 구현 완료 (2026-07-14)

사용자가 보여준 "뉴스ㆍ공시ㆍ재무" 통합 대시보드 스타일 중 **뉴스ㆍ재무 카드는
제외하고, 계열사 선택 pill + 공시현황만** 남긴 버전으로 축소해 구현. `공시`
페이지 최상단(기존 단판ㆍ공정위ㆍ지분 상세 탭 위)에 새 섹션을 추가:

- **계열사 선택 pill** — 주가현황 화면에 이미 나오는 5개 상장 계열사(동양ㆍ
  유진기업ㆍ유진투자증권ㆍYTNㆍ티엑스알로보틱스)를 pill 버튼으로 노출. 실제
  DART 공시 파싱이 동양(주) 하나만 구현돼 있으므로, 다른 계열사를 누르면
  "아직 연동되지 않았습니다"라고 정직하게 표시(가짜 데이터로 채우지 않음).
- **공시현황 통합 표** — 이미 서버가 갖고 있는 단판ㆍ공정위ㆍ지분(임원 소유상황
  보고ㆍ대량보유상황보고) 4개 데이터소스를 프런트에서 한 표로 합쳐 공시일
  최신순으로 정렬, 상위 20건만 표시(전체는 아래 상세 탭에서 확인). 유형별로
  배지 색을 다르게 표시(단판=보라, 공정위=주황, 지분(임원)=초록, 지분
  (대량보유)=분홍).
- 새 API 호출을 추가하지 않고, 공시 페이지 진입 시 4개 기존 라우트(danpan/
  ftc/equity/large_holding)를 병렬로 한 번만 불러와 재사용 — 서버 부하 증가
  없음.

이로써 사용자가 원래 요청했던 "뉴스ㆍ공시ㆍ재무" 포털 화면의 **공시 부분만
축소 구현 완료**. 뉴스ㆍ재무 카드, Supabase 이전, 별도 도메인 배포는 여전히
미착수(사용자가 "배포하진 않고"라고 명시했으므로 현재는 이 앱 위에서 실제
동작하는 형태로만 유지).

## 7. 독립 "공시 포털" 확정 (2026-07-13)

6번(기존 내부회계 포털에 얹는 방향)을 다시 뒤집었습니다. **기존 포털과는 완전히 별도인
"공시 포털"을 새로 만드는 것**으로 확정됐습니다. 이유와 바뀐 전제:

- **디자인 톤은 유지, 시스템은 분리**: 사용자가 보내준 스크린샷(증빙관리/KPI/AI챗봇)의
  색상ㆍ타이포ㆍ카드ㆍ표 스타일("저런식으로")은 그대로 참고하되, 실제 로그인/데이터/배포는
  기존 포털과 무관한 새 프로젝트로 만듭니다. 앞서 만든
  [공시관리 탭 목업](https://claude.ai/code/artifact/01efb895-7d7d-45c3-8ab9-80d207736832)이
  이 디자인 톤의 기준점입니다.
- **탭별 접근 제한 불필요**: 지분공시(임원 개인 매매내역)까지 포함해 이 포털에 들어갈
  내용은 전부 DART 등에 **이미 공개로 제출된 공시**라, 사업부서든 누구든 봐도 되는 정보로
  재정의됨. 그래서 6번에서 고민했던 Supabase RLS 기반 권한 분리, 사번 로그인 연동 자체가
  **필요 없어짐** — 배포가 훨씬 단순해집니다(로그인 없는 정적/공개 대시보드로 충분).
- **포털 구성 (3개 카테고리)**:
  1. **수시공시** — 지금의 단판공시(단일판매·공급계약)가 여기 속함. DART
     `pblntf_ty` 미지정(또는 'I' 거래소공시)으로 이미 조회 중.
  2. **공정위 공시** — 아직 미구현. DART Open API `pblntf_ty='J'`로 바로 조회 가능한 것을
     실제 확인함(별도 API나 스크래핑 불필요, 기존 방식 그대로 재사용). 동양(주) 최근 3년
     실제 내역으로 확인된 보고서 유형:
     - 대규모기업집단현황공시(분기별 공시 / 연1회공시 및 1분기용) — 계열회사ㆍ주식소유
       현황 등, 정기적으로 반복 제출
     - 특수관계인에대한출자
     - 특수관계인에대한채권매도
     - 동일인등출자계열회사와의상품ㆍ용역거래(변경)
     - 지급수단별ㆍ지급기간별지급금액및분쟁조정기구에관한사항
     - (전부 `report_nm` 텍스트로 구분 가능, 정정본은 기존 기능들과 동일하게 `[기재정정]`
       접두어로 판별)
  3. **지분공시** — 지금 만든 임원ㆍ주요주주 소유상황보고 + 대량보유상황보고서(5% Rule)
     그대로.
- **한눈에 보는 대시보드 + 점검 도구**: 세 카테고리를 한 화면에서 현황 요약으로 보여주고,
  그 안에서 개별 공시를 점검하고 **사전에 공시대상 여부를 판단하는 도구**(지금 단판공시에
  있는 사전검증 계산기 같은 것)를 카테고리별로 두는 것이 목표. 공정위 공시 쪽에도 비슷한
  사전판단 도구가 필요한지(예: 대규모내부거래는 자본총계 대비 일정 비율ㆍ50억원 이상 거래 시
  이사회 의결+공시 대상이 되는 공정거래법 기준이 있음 — 이런 기준을 계산기로 만들지)는 아직
  미확정, 사용자 확인 필요.

### 공정위 공시 사전검증 근거 (조사 완료, 2026-07-13)

독점규제 및 공정거래에 관한 법률 제26조(대규모내부거래의 이사회 의결 및 공시), 같은 법
시행령 제33조, 공정위 고시("대규모내부거래 등에 대한 이사회 의결 및 공시에 관한 규정")를
조사해 확인한 정확한 기준:

- **공통 거래금액 기준**: 거래금액이 **100억원 이상**이거나, 회사의 **자본총계 또는
  자본금 중 큰 금액의 5% 이상**(단, 그 5% 금액이 5억원 미만이면 **5억원**을 최소 기준으로
  적용) — 이 기준을 넘으면 이사회 의결 후 공시 대상.
- **거래유형별 차이** (사용자가 정확히 지적했던 부분):
  - 자금ㆍ유가증권ㆍ자산 거래: 특수관계인(동일인+동일인관련자+국내 계열회사 등, **국외
    계열회사는 제외**)이 상대방이면 위 금액기준으로 판단.
  - 상품ㆍ용역 거래: 위 금액기준을 충족해도, 거래상대방이 반드시 **"동일인 및 동일인
    친족이 발행주식총수의 20% 이상을 소유한 계열회사(A)" 또는 "A의 상법 제342조의2에
    따른 50% 초과 자회사(B)"**여야만 대상 — 일반 계열회사 전부가 아니라 이 좁은 범위로
    한정됨.
- **거래금액 산정**: 자금ㆍ유가증권ㆍ자산은 실제 거래금액(담보=담보한도액, 부동산
  임대차=연간임대료+환산보증금), 상품ㆍ용역은 **분기에 이루어질 매출ㆍ매입 합계액**.
- **공시시기**: 이사회 의결 후 상장법인 3영업일, 비상장/공익법인 7영업일 이내.

### 백엔드 구현 완료 (2026-07-13)

`server.py`에 기존 단판공시/지분공시와 동일한 패턴(DART Open API 조회 → 원문 파싱 →
집계 → 6시간 캐시)으로 두 기능을 추가하고 실제 동양(주) 데이터로 검증했습니다:

- **`fetch_ftc_monitoring()`** — `pblntf_ty='J'`(공정위공시)로 목록 조회 후 위 3종
  report_nm만 필터링(`[기재정정]` 접두어는 제외). 세 유형은 서식이 서로 달라
  (`_ftc_parse_invest`/`_ftc_parse_bond_sale`/`_ftc_parse_goods_services`) 각각 별도
  파서로 처리 — 전부 ACODE 없는 순수 라벨-값 `<TD>` 서식(단판공시와 유사)이라 라벨
  텍스트로 값을 찾는 `_ftc_value_for()` 헬퍼를 공용으로 씀. 실제 조회 결과 최근 10년간
  29건(특수관계인 출자 4건, 채권매도 2건, 상품ㆍ용역거래 23건) 확인.
  - 상품ㆍ용역거래 파싱 중 실제 버그 발견ㆍ수정: 이 서식은 "최초"(연 1회, 라벨-값이
    한 행에 나란히) 서식과 "변경"(분기별, 헤더 행 따로/값은 다음 행 같은 열) 서식이
    섞여 있는데, 처음엔 "변경" 서식에서 거래상대방 컬럼이 "매출액(B)" 같은 다른 헤더
    셀을 잘못 값으로 채택하는 문제가 있었음 — 같은 행 다음 셀 값이 "매출액"/"매입액"을
    포함하면 헤더로 판단해 다음 행 같은 열 위치로 대체하도록 수정.
  - `GET /data?section=ftc` 라우트.
- **`check_ftc_disclosure(transaction_type, amount, capital_base, is_goods_services_target)`**
  — 위 법령 기준을 그대로 구현한 사전검증 계산기. `transaction_type`은
  `'fund_securities_asset'`(자금ㆍ유가증권ㆍ자산) 또는 `'goods_services'`(상품ㆍ용역).
  `goods_services`인데 `is_goods_services_target=False`(거래상대방이 20% 요건 미충족)면
  금액과 무관하게 공시대상 아님으로 판단. 여러 시나리오로 로직 검증 완료.
  - `GET /data?section=ftc_check&transaction_type=...&amount=...&capital_base=...&is_goods_services_target=...` 라우트.

### 지분공시 3필드 추적 관리 기능 구현 완료 (2026-07-14)

DART 소유상황보고서 원문 XML을 직접 열어보니(`ACODE`/`AUNIT` 속성 확인)
3개 필드가 매 보고서마다 실려 있어서, 별도 외부 자료 없이 **보고서 간 대조만으로
불일치를 잡아낼 수 있다**는 걸 확인했다:
- `FLT_SUM`(발행주식총수) — 회사 전체 공통값, 매 보고서에 실림
- `STF_APT_DT`(선임일) — 임원 개인별, 매 보고서에 실림
- `BFR_PS_CPT_CNT`/`AFR_PS_CPT_CNT`/`MDF_PS_CPT_CNT`(직전/이번/증감 주식수) —
  "직전+증감=이번" 산식이 성립해야 하고, 이번 보고서의 "이번" 값이 다음 보고서의
  "직전" 값과 이어져야 함

`server.py`에 `_equity_compute_accuracy()`를 추가해 `fetch_equity_monitoring()`이
이미 파싱해둔 문서들로 3가지를 대조:
1. **선임일 일관성**: 같은 사람의 모든 보고서에서 선임일이 하나여야 함 → 다르면
   후보로 표시
2. **주식수 정합성**: 문서 내 산식 위반, 또는 보고서 간 연결 끊김 → 후보로 표시
3. **발행주식총수 타임라인**: 시간순으로 값이 바뀐 지점만 추려 노출, 같은 날짜에
   서로 다른 값이 보고되면 "충돌"로 별도 표시

`GET /data?section=equity_accuracy` 라우트로 노출하고, 지분공시 탭(임원ㆍ주요주주
소유상황보고서 이력 표 바로 아래)에 "지분공시 정확성 점검" 섹션을 추가해 3가지를
모두 보여줌. **실제 동양(주) 데이터로 즉시 실제 오류 후보 2건을 잡아냈다**:
- 이종면의 선임일이 한 보고서엔 "2025-05-01", 이후 보고서들엔 "2026-05-01"로
  다르게 기재됨(연도 오탈자 가능성)
- 2026.07.09 보고서 하나만 발행주식총수가 107,634,260(전후 보고서의 정확히
  절반인 215,268,520)로 기재됨(오탈자 또는 착오 가능성)

이 두 건은 자동 판정이 아니라 "확인 필요" 후보이므로, 화면에도 "사람이 다시

### 요청서류 체크리스트 기능 구현 완료 (2026-07-14)

신규 임원의 최초 소유상황보고와 기존 임원의 변동보고는 준비에 필요한 증빙이
다르다는 요청 사항을 그대로 구현:
- **신규**(최초): 임원 선임서(임원 명령서) + 매매거래내역증빙 + 잔고증명서
- **변동**: 매매거래내역서 + 잔고증명서

DART 소유상황보고서 원문에 이미 "보고구분"(신규/변동, `AUNIT="RPT_DST"`)이
실려 있어서 이것도 별도 자료 없이 바로 판별 가능했다. `_equity_parse_document()`가
이 값을 함께 뽑고, `fetch_equity_monitoring()`이 각 보유자의 **가장 최근 공시
건**의 보고구분을 기준으로 `required_documents`(요청서류 목록)를 계산해 API
응답에 포함한다.

지분공시 표(임원ㆍ주요주주 지분공시 이력)에 "요청서류" 열을 추가 — 각 행에
"신규 서류"/"변동 서류" 버튼이 있고, 누르면 해당 인원의 필요서류 체크리스트
모달이 뜬다. 서버 쪽에 회원 계정ㆍDB가 없는 소규모 내부툴이라, 체크(요청함/
받음) 상태는 브라우저 `localStorage`에 `rcept_no+문서명` 단위로 저장 — 팀
공유 PC라면 그 PC에서는 유지되지만 다른 PC/브라우저와는 동기화되지 않는다는
한계가 있음(추후 여러 명이 동시에 확인해야 하는 상황이 되면 서버 저장으로
옮겨야 함).

   노출하지 않음).
4. **요약 화면의 유형별 카드+필터는 유지**, 세부 탭과 완전히 분리된 화면

### "20% 계열사" 목록 18개 → 27개 보정 (2026-07-15)

사용자가 직접 파악한 실제 대상 계열회사 27개 목록을 주면서 왜 처음 뽑은

현황" 표는 **최근 3개년간 실제 거래 실적이 있는 회사만** 신고하는 표라서,
정의상 대상(20% 이상 소유 또는 그 자회사)이지만 그 기간에 거래가 전혀
없었던 회사 9곳(유진에너팜ㆍ유진엠플러스ㆍ유진소닉ㆍ하루테크ㆍ굿앤파트너스ㆍ
힐스프링인베스트먼트ㆍ자연팜앤바이오ㆍ경산기업ㆍ고운레저)이 누락됐던 것.

**해결**: 거래 실적표 대신 같은 공시의 "(1) 소유지분현황" 표(회사마다
동일인ㆍ친족ㆍ계열회사 주주구성 전부를 신고하는 표, 1,000행 넘는 ROWSPAN
중첩 표)에서 직접 계산하도록 로직을 새로 짬:
1. ROWSPAN이 수백 행까지 이어지는 표라 bs4가 생략한 셀을 못 채워주므로,
   행렬을 완전히 펴는 `_expand_rowspan_grid()`를 새로 작성(대기중인 셀을
   컬럼별로 추적하다가 남은 행 수만큼 채워 넣는 표준 rowspan 전개 알고리즘).
2. 회사별로 "동일인" 지분율 + "친족 합계" 지분율을 더해 20% 이상이면 A로
   분류.
3. A(또는 이미 확정된 자회사)가 50% 초과 보유한 계열회사를 반복적으로
   추가해 B를 계산 — 상법 제342조의2 제3항(자회사의 자회사도 모회사의
   자회사로 본다)에 따라 손자회사ㆍ증손회사까지 사슬로 확장되는 걸 반영
   (고정점에 도달할 때까지 반복).
4. "합자회사"(사모투자합자회사 등 지분(조합원권) 기반 법인)는 상법 342조의2가
   "발행주식" 기준이라 애초에 자회사 정의 대상이 아니므로 결과에서 제외 —
   실제로 유진더블유사모투자합자회사가 유진기업(주) 지분 56.98%를 보유해
   계산상 걸렸지만 사용자 목록엔 없어서 이 예외를 확인ㆍ반영함.

이 알고리즘으로 계산한 결과가 **사용자가 제시한 27개사와 정확히 일치**함을
확인(표기 형식만 "(주)X"/"X(주)" 혼용을 통일). `_group_status_parse_ownership()`ㆍ
`_compute_goods_services_targets()`로 구현하고 `fetch_goods_services_target_companies()`가
이 계산 결과를 반환하도록 교체 — API 응답 형태(`GET /data?section=goods_services_targets`)와
프런트엔드는 그대로 두고 내부 계산 로직만 바꿔서 드롭다운이 27개사 전부를
보여준다.

### 자본금 데이터 전체 계열사로 확장 (2026-07-15)

실제로는 판정이 가능해야 함 — 원문 매뉴얼 제9조의2②를 확인한 결과 근거
확보.

**핵심 발견(매뉴얼 및 실제 DART 원문 대조)**: "변경" 공시의 "2. 거래기간"
필드에 "4분기(2024.10.1~2024.12.31)"처럼 분기 종료일이 명시돼 있고, "8.
기타" 사유란에도 "실제 거래금액이 기 공시한 예상거래금액대비 20% 이상
감소함에 따른 변경 공시"라고 직접 명시돼 있음. 즉:
- **공시대상 확인**: 변경 공시가 존재한다는 사실 자체가 "20% 이상 감소"
  요건 충족의 근거이므로, 별도 금액 재계산 없이 항상 충족으로 표시.
- **공시기한**: 거래기간 필드에서 뽑은 분기 종료일 + 45일(역일, 영업일
  아님)을 기한으로 계산해 실제 공시일과 비교(고시 제9조의2②).

**예외 케이스 발견**: 2020년 일부 "변경" 건은 이사회 의결일이 실제로
기재돼 있었음(예: 2020.06.29 접수건, 사유가 "20% 이상 **초과**"로
증가형) — 매뉴얼에 "20% 이상 증가할 것으로 예상되는 경우에는 분기 중에
미리 이사회 의결을 거친 후 공시"라는 별도 단서가 있어, 감소형(사후공시ㆍ
45일)과는 절차 자체가 다름. 이사회의결일이 파싱되는 "변경" 건은 45일
규정을 적용하지 않고 확인불가로 표시하도록 분기 처리.

**옛 날짜 표기 버그**: 2020년 특수관계인출자(2020년 4월 28일 의결)ㆍ
채권매도(2020년 03월 09일 의결) 건이 "확인불가"로 나왔던 원인은 원문
이사회의결일이 "2020년 4월 28일"처럼 한글 날짜로 적혀 있어 기존
`_normalize_dot_date()`(점 구분 표기만 지원)가 파싱 못 했기 때문 —
한글 날짜 정규식을 추가해 해결(수정 후 두 건 모두 정상적으로 "준수"로
표시됨). 화면에 나가는 모든 날짜는 `_format_ftc_date()`로 "YYYY.MM.DD"
표기 통일.

**UI**: 공정위공시 탭의 "최근 1년"/"전체 이력" 버튼이 라벨 길이 차이로
한쪽만 줄바꿈되며 크기가 들쭉날쭉해 보이던 것을, 두 버튼 모두 "메인
라벨 + 부제" 2줄 고정 레이아웃(`#ftcRangeTabs .tab-btn`에 min-height +
flex column)으로 통일.

### 자회사 4곳 FTC 공시 통합 + &cr; 파싱 버그 수정 (2026-07-23)

사용자 피드백 두 가지를 반영:

1. **이사회 의결 있는 "변경" 건도 영업일 규정으로 검사**: 앞선 수정에서
   이사회의결일이 기재된 "변경" 건(증가형ㆍ사전 의결)을 무조건 확인불가로
   뒀는데, 사용자가 "이사회 결의 필요한 부분은 이사회 했잖아, 이사회
   의결일로부터 3영업일 내로 공시했는지 체크해주면 돼"라고 지적 — 이제
   이런 건은 다른 이사회의결 기반 유형과 동일하게 영업일 계산 경로를
   탄다(`_ftc_apply_accuracy_check`에서 `has_board_date` 분기). 참고: 과거
   (2020~2021년경) 영업일 기준이 지금(3영업일)과 달랐을 가능성을 사용자가
   제기했으나, 공정위 고시 개정 연혁에서 정확한 시점을 웹검색으로도 확인하지
   못해 전 기간 현재 기준(3ㆍ7영업일)을 동일 적용함 — 명확한 자료가 나오면
   재조정 필요.
2. **자회사 4곳(동양에너지ㆍ금왕에프원ㆍ디씨아이티와이부천피에프브이ㆍ
   디씨아이티와이인천피에프브이)의 공정위 공시도 통합**: 각 회사 이름으로
   개별적으로 찾아야 하는 번거로움을 없애기 위해 `FTC_MONITORED_ENTITIES`에
   corp_code를 추가해 5개 회사분을 한 번에 조회ㆍ병합하도록 확장(제출회사
   컬럼 추가, 공시기한은 제출회사가 상장(동양(주))이면 3영업일ㆍ비상장
   (자회사 4곳)이면 7영업일로 분리 적용). 이 과정에서 금왕에프원(주)이 내는
   4번째 보고서 유형("특수관계인으로부터자금차입")을 새로 발견해 파서 추가.
   동양에너지(주)는 동명이인 회사가 있어 DART corpCode.xml 마스터ㆍ주소ㆍ
   대규모기업집단현황공시 제출 이력으로 교차 확인 후 corp_code 확정
   (00935609, 여의도 소재 — 신안군 소재의 별개 동양에너지(00672348)와 구분).

**버그 발견 및 수정**: 사용자가 2019.12.24ㆍ2020.06.29ㆍ2020.12.23ㆍ
2021.12.16 네 건을 재검증해달라고 요청 — 확인해보니 이 옛 공시 문서들은
라벨 셀 안에서 줄바꿈을 "&cr;"라는 비표준 HTML 엔티티로 넣는데(예:
"이사회&cr;의결일"), BeautifulSoup이 이를 인식 못 해 리터럴 문자열로 남아
`_ftc_value_for()`의 라벨 부분일치가 깨졌던 게 이 네 건 모두(및 다른 여러
과거 기록)에서 이사회의결일ㆍ매출액대비 비율이 빈 값으로 나온 근본 원인이었음.
`_ftc_value_for`에서 "&cr;" 제거 후 비교하도록 수정하고, 매출액대비 값에
"%" 기호가 없는 옛 서식(정규식으로 못 잡힘)을 위해 `_expand_rowspan_grid()`
기반 열 위치 폴백(`_ftc_parse_goods_services_ratios_by_grid`)을 추가함.
수정 후 네 건 전부 정상적으로 이사회의결일ㆍ영업일수ㆍ공시대상 재확인이
계산되며 "준수"로 표시됨을 확인 — 전체 33건 중 filing_timeliness가
"unknown"으로 남은 건이 0건이 됨.

### 공휴일 미반영 영업일 버그 + 다분기 묶음 문서 비율 오매치 수정 (2026-07-23)

사용자가 금왕에프원(주)의 2026.01.09 자금차입 건이 "⚠지연 의심(8영업일)"로
잘못 표시된 걸 지적 — 이사회 의결(2025.12.30)부터 공시(2026.01.09) 사이에
신정(1/1)이 껴 있는데 `_business_days_between()`이 토ㆍ일만 빼고 공휴일은
전혀 고려하지 않아서 실제로는 7영업일인데 8영업일로 잘못 셈. 매년 날짜가
고정된 공휴일(신정ㆍ삼일절ㆍ어린이날ㆍ현충일ㆍ광복절ㆍ개천절ㆍ한글날ㆍ성탄절)을
`_FTC_FIXED_HOLIDAYS_MMDD`에 추가해 영업일 계산에서 제외하도록 수정 —
수정 후 정상적으로 7영업일ㆍ준수로 표시됨. 설날ㆍ추석ㆍ부처님오신날은 음력
기준이라 연도별 조회표 없이는 계산 불가능하고, 대체공휴일도 아직 반영 안
함 — 여전히 남은 한계.

같은 요청에서 "2020.06.29 상품ㆍ용역거래 변경" 건과 "2019.12.24 공시"가
"공시대상 재확인"에서 왜 "미달"로 나오는지 문의 — 조사 중 실제 버그를
하나 더 발견함: 2020.06.29 건은 한 공시 문서 안에 2020년 2/4ㆍ3/4ㆍ4/4분기
세 개 분기 블록이 통째로 묶여 있는데(증가형 변경이라 이사회에서 세 분기를
한꺼번에 미리 의결), 매출액대비 비율을 뽑는 정규식이 그냥 "거래금액"이라는
문자열을 앵커로 썼더니, 이 문자열이 "5. 상품ㆍ용역 거래내역" 표의 열
헤더로도 또 등장해서 다음 분기 블록의 "8. 기타" 사유문("...20%이상
초과하여 변경함")까지 통째로 삼켜버려 엉뚱하게 "20.00%"를 최대 비율로
잘못 인식하고 있었음. 앵커를 "동일인 등 출자 계열회사와의 상품ㆍ용역
거래금액"이라는 유일한 전체 라벨로 바꿔 수정 — 실제 최댓값인 2.78%로
정정됨. (2019.12.24 건은 이 버그와 무관 — 아래 참고.)

**"미달"은 버그가 아님 — 실제 숫자로 확인**: 동양(주)의 현재(2026년) 자본총계
는 6,981억원(5% = 349억원)인데, 2020.06.29 건의 거래금액은 약 99.87억원,
2019.12.24 건은 약 59.99억원으로 둘 다 100억원ㆍ현재 기준 5% 모두 못
미친다. 이건 화면에 이미 적어둔 안내(신고 당시가 아닌 현재 자본 기준이라
참고용)가 실제로 딱 들어맞는 사례 — 2019~2020년 당시 동양(주) 자본총계는
지금보다 훨씬 작았을 것이므로 그 시점 기준 5%는 더 낮았고, 그래서 그때는
정상적으로 공시대상이었을 가능성이 높다. 코드 수정 대상이 아니라, 자본
성장에 따른 자연스러운 결과.
